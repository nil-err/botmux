# Dashboard UI Baseline

本文件是 botmux dashboard 后续 UI 重构的设计基准。新增页面、重构页面、调整组件样式前，先对齐这里的规范；如果出现新的共识，先更新本文档，再落到代码里。

## 目标

- 面向开发者和运维使用场景，优先保证信息密度、可扫读性、状态清晰和操作稳定。
- 组件风格要可复用，页面重构时优先沉淀公共结构，而不是每页单独写一套样式。

## 布局骨架

- 顶栏横向贯穿全宽，承载品牌、低频工具入口和 owner 头像。
- 左侧导航是独立浮动纵向 rail；桌面端始终展示 icon + label，移动端转为横向滚动导航。
- 内容区保持高密度后台布局，页面主体不做营销式 hero。
- 顶栏和导航只放全局入口，页面级主操作、筛选、批量操作、二级 tab 放在页面内部。
- 页面和内部列表的滚动条使用悬浮式视觉层，不占布局宽度；不要通过 `scrollbar-gutter` 或额外右侧留白预留滚动槽，避免内容在出现滚动条时被挤压。

## 圆角

```css
--radius-xs: 12px;
--radius-sm: 12px;
--radius-md: 12px;
--radius-lg: 12px;
--radius-xl: 12px;
--radius-full: 999px;  /* 头像、状态点、明确的 pill */
```

使用规则：

- 普通矩形容器、卡片、列表项、输入框、选择器、导航项统一使用 `12px`。
- 页面内文本按钮统一使用胶囊圆角 `999px`；头像、状态点、真正的 pill 也用 `999px` 或 `50%`。
- 不再使用按组件层级区分的多档圆角；保留多组 token 只是兼容旧选择器，实际值保持一致。
- 不要把普通矩形容器做成 `999px`；胶囊圆角只用于文本按钮、短 pill、头像和状态点。

## 顶栏

- 左侧：logo、`Botmux`、`Dashboard`。
- 右侧：语言、主题、Docs 等低频工具和 owner 头像。
- 创建会话、添加机器人等页面级主操作放回对应页面标题区，不放在顶栏。
- 会话概览是顶栏唯一的全局会话状态入口；待处理通知收进会话概览弹窗顶部的紧凑 notice，不单独在顶栏中间占位。
- 当待处理数量从 `0` 变为非 `0` 时，会话概览可自动展开约 4 秒作为轻量提醒；用户 hover/focus 时继续保持展开，主题菜单打开时不自动抢占弹层。
- 低频工具使用独立圆形图标按钮，边框和背景弱化。
- owner 头像固定在最右侧，作为身份和全局账户入口的视觉锚点。
- `Botmux` 与 `Dashboard` 品牌文字共享 logo 式掠光：保留各自基础字色，只让一条窄的平台色高光以约 8 秒线性扫过；首帧高光须在可视区外，不能做成整段金属渐变、持续闪烁或 hover 才触发的高频动效，`prefers-reduced-motion` 下关闭。

## 导航

- 桌面端左侧浮动 rail 保持稳定宽度，显示 icon + label。
- 不提供只显示 icon 的折叠态，避免语义依赖 tooltip 或记忆。
- 当前页高亮使用轻量背景 + 细边框，不使用大面积强色块。
- 导航项数量较多时允许 rail 内部滚动，不挤压内容区。
- 移动端导航改为横向滚动 tab，不做抽屉导航，避免多一步操作。

## 按钮和控件

- 工作台是按钮和布局的视觉基准：`page-heading-actions`、`SectionHeader` 右侧操作、列表项内短操作按钮应和工作台的“查看全部 / 立即处理 / 排序”保持同一高度、字重、边框和 hover 反馈。
- 页面内文本按钮统一使用胶囊风格：高度使用 `--button-height`，圆角使用 `--radius-full`，横向内边距使用 `--button-padding-x`。
- 页面标题右侧动作按钮和标题下 toolbar 使用同一套胶囊尺寸：34px 高度、12px 加粗文字、`--button-padding-x` 横向留白；常规动作使用普通镂空按钮，刷新/创建这类强调动作使用平台色镂空按钮。
- 胶囊内文字的主横向留白按纯视觉标准统一，不按功能语义区分；常规文本按钮、搜索框、筛选 chip、状态 chip、批处理 chip 默认使用 `--button-padding-x`（当前 15px）。带 chevron 的下拉可额外预留箭头空间，带 switch 的控件可额外预留滑块空间，但文字起始留白仍对齐 `--button-padding-x`。
- 同一 toolbar 内的同等级文字动作按钮默认不加额外固定宽度，靠内容宽度 + `--button-padding-x` 形成胶囊；只有同组短状态/分段控件需要稳定宽度时才单独指定。
- 页面标题下面的顶层筛选/导航栏统一使用 `dashboard-toolbar`：透明背景、无外框面板、8px 控件间距、控件高度 34px。不要恢复成带边框的大面板式 `.filters`。
- 标题下 toolbar 的单选下拉筛选优先使用 React `DropdownMenu` 自定义菜单，保证 trigger、弹层、选项 hover/选中态一致；不要在这类位置继续使用原生 `select`。
- 页面内一级 tab 切换使用紧凑 segmented rail：外框 34px、内距 2px、选中块 28px、灰色选中背景 + 平台紫色文字；不要使用旧式上圆角 tab + 底线组合。
- 团队协作、角色管理这类页面级二级入口统一使用 `insight-tabs` / `itab` 结构的 segmented rail，当前项用 `on` / `aria-selected` 表达；不要混用 `dashboard-toolbar` 链接和单独的 `active` 样式。
- 一级 tab 的页面状态/加载计数放在 tab rail 右侧同一行，作为弱化文本处理；不要单独占一行挤压内容区。
- 会话控制页内的分段 switch 选中态统一为灰色背景 + 平台紫色文字，包括工作流/团队/机器人、看板/状态板/表格、24H/72H/7d 这类同组切换；不要混用黑底白字、紫底白字或发光选中态。
- 条件分组控件按可见性展开/收起时使用约 320ms 的缓入缓出，不做突然消失或快速跳变；`prefers-reduced-motion` 下不保留过渡。
- 页面标题下面的分段 switch 尺寸以 24H/72H/7d 阈值组为基准：外框高度 34px、内距 2px、选中块高度 28px，选中背景到外框四周留白一致。
- 带静态条件名的 switch 需要组合成同一个胶囊控件，例如“空闲超过 + 24H/72H/7d”；不要把条件名和取值拆成两个并列 chip。
- “全部 + 若干同类枚举值”的顶层筛选也按同一个 segmented switch 呈现，例如洞察页的“全部 / claude-code / codex”；不要拆成多个独立 chip。
- 会话列表和详情内的短统计/状态 tag 使用平台胶囊 tag：弱边框、`--surface` 背景、`--radius-full`、12px 加粗文字；异常 tag 使用对应语义色的弱背景和边框，不用左侧预留色条制造空白。
- 会话控制这类横向筛选 toolbar 里的搜索框也按胶囊控件处理，不能继承通用 input 的 12px 圆角和 11px 留白；新增筛选输入框时必须显式落到 toolbar 的 pill input 规则，并让 hover/focus 反馈对齐同组下拉框。输入型胶囊 hover/focus 使用同一个平台色边框和浅平台色背景，不加外层模糊 focus ring。
- 创建 / 新增类动作默认使用平台色镂空胶囊按钮，文案左侧使用统一的 `+` 图标；页面级创建、modal 内创建提交保持同一视觉层级。典型入口包括创建会话、添加机器人、新建群、创建 Webhook、创建团队。
- 所有常规文字按钮必须直接使用 `--button-padding-x`，不按“创建、保存、查看、清理”等功能另设横向内边距。例外仅限圆形图标按钮、分段控件和卡片内的紧凑操作；这些控件按自身固定几何尺寸定义，不与文本按钮混用。
- 刷新入口统一使用 34px 的圆形回环箭头图标按钮，不保留可见文案；必须提供 `aria-label` 与 `title` 作为可访问名称和 tooltip。
- 主按钮：仅用于不可替代的最终确认或强状态动作，使用平台色实心胶囊按钮，不做强泛光。
- 次级按钮：透明或弱背景，使用 accent 边框/文字，仍保持胶囊形态。
- 危险/清理类 toolbar 按钮仍使用镂空胶囊，但默认边框不能低透明到发灰；保持清晰危险色描边，hover 只加弱危险色背景。
- 普通工具按钮：低对比边框、hover 时轻微高亮；不要在不同页面用不同高度、不同字重。
- 下拉框以工作台排序菜单为基准：单行 `select` 和自定义 dropdown trigger 使用 `--button-height`、`--radius-full`、`--surface`、`--border-soft`，文字紧凑加粗；弹层使用 `--surface`、`--border-soft`、`--radius-lg`，选项为胶囊行，默认透明边框，hover 使用 6% accent 背景，选中使用 8% accent 背景和更明显的 accent 边框。
- 短选项下拉必须使用稳定宽度，不能随当前文案自动伸缩；默认触发器宽度使用 `--dropdown-trigger-width`（当前 118px），同一控件组内的状态、接入、CLI、团队等下拉保持相同宽度。弹层最小宽度不得小于触发器；如果弹层内容需要更多空间，可以只放宽弹层，不放宽触发器。
- 单行下拉 hover/focus 不改变布局尺寸；自定义下拉必须支持点击外部和 Escape 关闭。`select[multiple]` 是多行列表例外，只统一 surface、弱边框、12px 圆角和 hover/focus，不做单行胶囊。
- Dashboard 弹窗必须使用 React 组件实现；禁止新增 `innerHTML`、手动 `document.createElement('dialog')`、手写 DOM 事件拼装的 modal。旧功能迁移时只保留必要的打开入口，真实弹窗状态、表单、轮询和提交逻辑都放到 React 组件内。
- Dashboard modal 的按钮、短输入、select、多选/单选项统一使用胶囊风格：高度 `--button-height`，圆角 `--radius-full`，间距 8px/12px，原生 checkbox/radio 不直接露出。
- Dashboard modal 的表单字段按整列阅读，单行输入和自定义下拉默认撑满字段宽度；toolbar 里的短筛选下拉才使用 `--dropdown-trigger-width` 这种稳定短宽度。
- 配置型页面的字段解释不占用底部说明行；短说明放在字段标题或小节标题右侧的圆形 `?` 提示点里，hover/focus 时展示 tooltip，默认页面只保留标题和控件。
- Dashboard modal 右下角确认/取消成对出现时，按钮组右对齐，取消/关闭放在内侧（确认按钮左边），确认/提交/危险确认放在外侧最右。确认类按钮默认使用平台色镂空胶囊；危险确认使用危险色镂空胶囊，不使用实心按钮。
- 创建会话 modal 使用两级间距：一级模块之间统一 12px，控件内部、按钮组、选项列表统一 8px；不要用额外 margin 叠加间距。modal 内所有小标题统一使用 fieldset/legend 的嵌入式标题。
- 创建会话 modal 的“更多选项”里，群名称、工作目录这类短文本配置项按 pill 输入框处理：高度使用 `--button-height`，圆角使用 `--radius-full`，字段内部 label/input 间距 8px，不要复用通用 `.form-row` 的普通输入框圆角。
- 创建会话 modal 的机器人、协作模式、入列选项都使用真正的胶囊选项；不要在胶囊里显示原生 checkbox/radio，选中态由胶囊边框、弱背景和状态点表达。机器人选项按内容自适应宽度排列，只保留必要最小宽度，不要拉伸成整列宽块。
- 创建会话 modal 的 Lead 机器人 select 也属于胶囊控件：高度、圆角、边框、hover/focus 必须和短文本输入框一致。
- 创建会话 modal 的可选配置使用嵌入 legend 的轻量折叠按钮，默认收起；不要恢复成小胶囊下拉按钮。字段仍包在独立 grid 容器里，展开态使用 12px 模块间距。
- 创建会话 modal 的标题居中展示，不使用 section title 的圆点前缀。
- 创建会话 modal 标题区不使用底部分割线，靠固定间距和内容块边界建立层级。
- 新建群聊 modal 跟随创建会话 modal 的视觉规范：标题居中，机器人、群名、工作目录、Profile 直接展示，输入框、自定义下拉、机器人选项和底部按钮全部使用胶囊规范。
- 例外：纯图标按钮、头像、状态点、终端分段按钮可以保留圆形或自身结构，但不要和文本按钮混用视觉层级。
- 图标按钮优先使用 icon，文本只保留在语义不清或核心命令上。
- 相同功能在不同页面中必须使用同一视觉层级。

## 卡片和面板

- 卡片只用于独立信息块、重复项、弹窗和工具容器。
- 不做卡片套卡片；页面 section 不要为了装饰全部卡片化。
- 卡片标题区应紧凑，操作按钮放右上角或标题行末尾。
- 空态、错误态、加载态要有固定高度或最小高度，避免页面跳动。
- Dashboard 数据加载态统一使用 `LoadingState` / `.page-loading` 的旋转环：平台色 top border、0.8s 线性旋转，并在对应组件容器内水平垂直居中。页面级加载放在页面内容区，卡片/弹窗/列表加载放在当前卡片、弹窗或列表区域内；不要新增裸 `加载中...` 文本、旧 `loadingHtml()` 或局部自绘 spinner。

## 列表

- Dashboard 内的紧凑列表默认使用“每项独立边框”的行卡样式，而不是只用底部分割线。
- 列表容器使用内边距和固定间距组织列表项；列表项使用 `--radius-lg`、`1px` 弱边框和轻量背景。
- 同一页面内功能相近的列表必须复用同一套列表结构。主标题/副标题放左侧，状态、时间或数量放右侧。
- 列表项 hover 使用轻微背景变化、accent 弱边框和最多 `1px` 的上浮反馈；不使用强色块或大阴影。
- 非可点击列表项 hover 不改变鼠标为 pointer；只有真实链接、按钮或可进入详情的行才使用 pointer。
- 容易横向溢出的单行标题、群组名、路径摘要默认使用省略号；需要 hover 展示完整内容或横向滚动时，统一使用 `OverflowText`。组件只在真实溢出时启用平台风格浮层和横向滚动；未溢出的文本不要额外弹 tooltip，也不要使用浏览器原生 `title` 充当主展示能力。
- 空态可以放在同一列表容器内，但要保持和列表项相近的内边距与高度，避免出现空白塌陷。
- 列表区需要排序或筛选时，控件放在标题行右侧，并位于“查看全部”等跳转按钮左侧；排序状态要用明确文本展示当前模式。
- 群组管理这类“对象 x 覆盖状态”的页面默认按工作台活跃会话列表组织：对象信息在左，覆盖状态用行内平台 tag/chip 展示，操作按钮放右侧；不要回退成普通矩阵表格。

## React 组件

- 新页面优先使用 `src/dashboard/web/dashboard-components.tsx` 里的基础组件，不在页面内复制同类 DOM 结构。
- `SectionHeader` 用于页面 section 标题行；标题左侧圆点、计数 tag、说明、右侧操作都走同一结构。
- `HeaderAction` 用于“查看全部”等标题行次级跳转按钮；多个控件时用 `HeaderControls` 包裹。
- `OverviewList`、`OverviewListItem`、`OverviewListMain`、`OverviewListTail` 用于工作台同类紧凑列表。列表项高度由内容决定，不因列表容器变高而拉伸。
- `DropdownMenu` 用于短选项下拉筛选/排序；`SortMenu` 是排序语义的兼容封装。触发按钮只展示当前模式，不展示“排序”前缀；触发器宽度默认使用 `--dropdown-trigger-width`，选中项和 hover 项使用不同背景，避免状态黏连。
- `LoadingState` 用于所有数据加载占位；页面级使用默认尺寸，卡片、弹窗、列表、局部面板使用 `compact` 或页面 class 缩小高度，但必须复用同一个 `.page-loading-spin`。
- `OverflowText` 用于单行文本溢出处理：外层容器必须有明确宽度或 `min-width: 0`，内部文本由组件检测 `scrollWidth`，只有溢出时才显示平台风格 fixed tooltip 并执行横向 pan 动画。不要在各页面复制测量、portal tooltip 或 marquee 逻辑。
- 公共组件只沉淀结构和状态 class，不写业务数据逻辑；页面自己的 API、过滤、排序、权限判断保留在页面模块内。

## 页面规范化

- 每个 dashboard route 的根节点使用 `page` 加页面语义 class，例如 `sessions-page`、`groups-page`、`connectors-page`、`roles-page`、`workflows-page`。
- 页面标题统一使用 `page-heading`：标题说明放在标题右侧，并与标题底部对齐；窄屏再回落为上下排列。右侧只放本页面主操作和当前页面视图切换，刷新、创建、导入等页面级动作不要塞进筛选条。
- Section 标题说明同样放在标题右侧底部对齐；说明文本不要独占标题下方空间，除非是移动端换行回退。
- 所有页面 title 位置以工作台为基准：页面主标题使用同一标题行高度和底部分隔线间距；section 标题使用工作台 `SectionHeader` 的 36px 标题行、左侧圆点、8px 标题到内容间距。
- 页面级筛选统一使用 `filters`，视觉上是紧凑工具条，不再做厚重卡片。
- 页面内容按工作台方式拆成 `overview-block` 式 section：先放 section 标题，再放列表、面板或表格；不要让卡片内部标题、外部标题和按钮层级混在一起。
- 设置/配置类页面不要把所有表单塞进一个大背景容器；按访问、投递、维护、版本等职责拆成独立模块卡片，模块之间用页面网格建立关系。
- 页面内容区的普通容器统一使用 `card`、`bd-card`、`panel` 或对应页面已有 panel class；默认实色背景、弱边框、12px 圆角、不使用玻璃泛光。
- 表格、矩阵、列表项、树节点、技能卡、Webhook item、Workflow panel 都按“独立边框行卡”处理；hover 只做轻量背景和边框反馈。
- Webhook、团队协作这类对象列表默认使用“标题 + 状态 tag + 元信息 tag + 右侧操作”的行卡结构；长 URL、邀请码、curl 示例放在行卡内部的弱背景代码块里，不要回退成裸表格或长串内联文本。
- 内联样式只允许表达一次性布局或运行态数值；背景、圆角、边框、hover、状态色应沉淀到 CSS class。

## 色彩和动效

- 默认主题主色从 logo 主色提取，使用 `#646CEA` 作为 `--brand-accent` 的品牌基准；明亮模式可用略深的 `--accent`（当前 `#565EE4`）承载小号文字按钮，确保在页面背景和白色卡片上都达到 WCAG AA 普通文本 `4.5:1`。
- 暗色主题不要使用近黑背景 + 近白文字的极端组合；主文字/背景对比保持清晰但克制，目标约 `10:1` 到 `13:1`，避免大面积 `17:1+` 造成阅读疲劳。次级文本仍需满足普通文本 `4.5:1`。
- `--faint` 只用于弱提示、禁用态、辅助图标和装饰信息；默认对背景至少接近 `3:1`，不要承载正文、按钮文字或关键状态。
- 普通装饰边框可以低于 `3:1` 以保持界面轻量；输入框、按钮、当前选中、危险/确认等关键控件必须通过文字、背景、边框或状态组合清晰可辨。
- logo 亮部青色和粉色只用于品牌字标或少量强调，不作为全平台主操作色。
- 默认主题保持深色、克制、开发者工具感；页面背景使用实色 `--bg`，不加装饰性渐变、极光或点阵纹理。
- accent 用于状态、主操作、当前选中，不做全页单一色彩统治。
- 动效只用于反馈和状态变化，避免长期循环动画分散注意力。
- 所有 dashboard route 的 `.page` 根节点使用统一进入动效：0.35s 淡入 + 6px 上移回正；侧边栏切换页面时只让内容区进入，顶栏和导航不参与。必须尊重 `prefers-reduced-motion: reduce` 并关闭该动效。
- 页面内部会替换大块内容的 switch/tab，例如会话控制的视图模式和看板分组，也使用同一套 0.35s 内容进入动效；只作用于被替换的内容区，不让标题、筛选栏和批量操作栏一起闪动。
- 顶栏“会话概览”这类关键 hover/focus 弹窗使用同方向的轻量 popover 进入动效；不扩大 hover 触发范围，不和主题菜单等其它顶栏弹层同时响应。
- 参考 ReactBits 的微交互和层次感，但不要把 dashboard 变成展示型页面。

## 字体和密度

- 不用 viewport width 缩放字体。
- 顶栏、导航、表格、卡片内标题保持紧凑字号。
- 只有页面主标题可以使用较大字号。
- 后台页面优先让用户扫描、比较和重复操作，不追求大幅留白。

## 响应式

- 桌面：顶栏全宽，左侧 rail + 右侧内容。
- 平板/手机：顶栏允许分行，导航变横向滚动。
- 移动端顶部横向导航只保留页面入口；创建会话、创建机器人等动作回到对应页面标题区，不在导航上方重复展示。
- 按钮文字不能溢出容器；必要时换行或隐藏低优先级上下文。
- 移动端优先保留核心操作，不隐藏创建会话和当前导航。

## 更新流程

- 页面重构前先确认是否已有对应规范；没有则先补本文档。
- 页面重构目标是直接收敛到 React 实现，不保留“React 壳 + legacy controller”的双实现过渡态。
- 旧的 DOM 字符串/imperative 渲染实现要删除；确实还被复用的代码只能以纯数据类型、API helper、格式化 helper 的形式保留。
- React 迁移不能丢功能：API、路由 hash、表单字段、筛选/排序、权限提示、空态、错误态、轮询、拖拽、快捷键、modal 等原有行为必须等价迁移。
- 新增公共组件前，先判断它属于按钮、导航、卡片、表单、状态、弹窗哪一类。
- 如果某个页面需要打破规范，要在 PR 描述里说明原因和影响范围。
- UI PR 应附截图，至少包含桌面和移动端。
