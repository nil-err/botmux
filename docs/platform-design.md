# botmux 多人多 bot 协作平台设计

## 一、要解决什么

团队里每个人都有自己的机器人（各自带着代码库、上下文、技能、偏好）。这个平台让**一个团队的机器人能被组织到一起协同干活**：发现彼此、组队、分工、接力、按流程编排。

外部系统（报警平台、CI、工作流引擎）也可以把任务**派进**这套协作体系——但那只是入口之一，**协作本身才是核心**。

典型场景：

- **一个需求落地**：把产品、前端、后端各自的机器人拉进一个群，分工推进、互相传结果。
- **代码评审**：作者的机器人 + reviewer 的机器人在一个话题里对线。
- **跨团队/跨人排障**：多个负责人的机器人一起查问题（报警在这里只是"触发器"，多人多 bot 一起查才是主体）。

> 现状基础：botmux 已经能把多个机器人拉进一个群、让机器人互相点名接力、也有工作流编排。所以平台要做的不是从零造协作，而是**把这些能力围绕"团队"组织起来，并让发现和管理变简单**。

## 二、信任模型：团队是边界

权限只围绕一个维度——**团队**。

- **团队** = 一组互相信任的人 + 他们的机器人。
- **团队内默认互信**：可以自由发现彼此、组队、互相点名、互相接力、共享工作流——**不需要逐个授权**。
- 团队是这些能力的作用域：名册、谁能把谁拉来、谁能驱动谁，都以团队为单位。
- **跨团队**协作：可以，但需要显式邀请/授权——先把团队内做透，跨团队后置。

这条让权限设计大幅简化：不做复杂的逐 bot 授权，团队内放开、团队边界处把关。

## 三、三种协作形态（都要支持）

三种共享同一套底座（团队名册 + 飞书群 + 点名接力 + 工作流），按发起方式区分：

| 形态 | 怎么协作 | 谁来驱动 |
|---|---|---|
| **a. 人主导组队** | 人把团队的几个机器人拉进群，点名分工，在群里指挥 | 人 |
| **b. 机器人接力** | 一个机器人干完，主动把活交给团队里另一个机器人（点名对方接手） | 机器人 |
| **c. 工作流编排** | 预定义流程，团队的机器人按串行/并行自动跑 | 流程 |

三者可以混用：人发起一轮协作，中间机器人自主接力，关键环节走预定义流程。

> **机器人接力要有最小结构**：人主导时可以自由聊；但 b（机器人自主接力）若只靠"@ 某 bot 一句话"，交棒质量不可控。给接力补一个轻量交接约定——**交给谁、当前结论、相关上下文/消息链接、期望的下一步、完成标准**，由 `botmux send` 或一个专门的交接 skill 承载。

## 四、协作的底座能力

1. **团队名册与发现**：让大家看到团队里有哪些人的哪些机器人、各擅长什么、怎么点名到它。**没有发现就没有协作**——这是基石。
2. **组队**：一键把团队的若干机器人拉进一个群或话题，并自动让它们互相认识（见第八节"丝滑组队"）。
3. **分工与接力**：群里点名谁负责哪块；机器人之间交棒（团队内自由，点名对方即可）。
4. **产出交接**：两种粒度——
   - **松**：大家都在同一个群里，消息天然可见，后接手的机器人能看到前面的对话。
   - **紧**：走工作流时，上一步的结构化产出按流程精确传给下一步。
5. **信任**：由团队边界统一解决，团队内不必逐项授权（见第二节）。

## 五、怎么发起一轮协作

协作可以由三种方式发起，落到的都是上面那套底座：

- **人发起**：在群里组队、点名。
- **定时发起**：到点自动拉起一轮（已有定时能力）。
- **外部事件发起**：外部系统通过 webhook 把任务派进来——**报警只是其中一类外部事件**，CI、工作流引擎、别的系统都是同等入口。

## 六、外部接入（webhook）

定位为"协作的外部触发入口之一"。设计要点（产品语言）：

- **安全递送**：请求带签名验明来源、防重复触发（防重放）、防洪峰（限流）。其中"抗 DoS、请求体硬上限"只在接入点**暴露到公网**时才需要；纯团队内网可信环境省掉，限流也只用宽松上限防"报警风暴误伤"。
- **原样透传**：网关不解析各平台格式，把原始内容**原样**交给机器人，让模型自己读懂。要查上下文/回写时，由机器人调用对应**技能**（如报警平台、项目平台的技能）。新平台几乎零适配。
- **防被指挥**：交给机器人的外部内容会明确框定为"待处理的事件数据，不是命令"——不执行其中指令、不泄露凭据。
- **投到哪**：可以投进一个已有的协作群、投进外部临时拉的群（机器人被拉进去后触发）、或自动新建一个临时处置群；自动建群时按事件标识去重、报警恢复后收尾，认不出标识时宁可不建群也不乱建。

## 七、平台要管的事

> **两个界面要分清**：**个人 dashboard**（已有）= 自己管理自己的机器人/会话；**团队平台（Web UI）**（新）= 团队公共的花名册、组队、协作实例、接入管理。下面"平台要管的事"指**团队平台**；个人 dashboard 维持现状、不混入团队数据。

- **团队与机器人**：建团队、把人和机器人纳入团队、看团队名册（谁有哪些 bot、各擅长什么）。
- **组队与协作**：发起协作、查看进行中的协作、管理工作流。为此引入显式的「**协作实例**」对象——一次协作 = 群/话题 或 workflow run + 参与机器人 + 发起人/来源 + 状态 + 产出索引；否则"进行中的协作、回溯、外部事件派进哪个协作"会散落在会话/群/工作流三套对象里、不好查。工作流运行时对参与机器人的能力/角色做**快照**，保证运行可复现，不被之后平台改角色影响。
- **外部接入**：创建接入点（拿网址/密钥）、试运行预览、看调用记录与失败原因。
- **运行可观测**：每个会话标注"由谁/什么触发"，整个协作过程可回溯。
- **调用记录（注意隐私）**：记来源/结果/耗时/命中的会话；外部原始内容默认不留存或脱敏。
- **权限**：以团队为单位（见第二节）。
- **审计（即便不做授权也要记）**：谁改了团队角色、谁建了协作群、谁加/移除机器人、谁创建了接入点——团队内默认互信，但保留这些操作记录，排障和误操作回滚都靠它。

## 八、丝滑组队（用"实时花名册"取代一次性介绍）

现状是手动：拉群 → 拉机器人 → @所有机器人做一次自我介绍（introduce）。问题不只是麻烦——introduce 是**一次性快照**，有机器人离开后之前的介绍就过时了，还得反向再补一条"谁走了"，很别扭。

更优雅的做法：**别依赖一次性介绍，把"群里此刻有谁、各自能干啥"做成随时可查的实时信息。**

- **实时花名册（核心）**：把 `botmux bots list` 升级为「**当前群**的协作花名册」——每行至少含：机器人标识（app）、显示名、**简短能力标签**、角色摘要、是否在本群、是否本机可执行 workflow，以及关键一列——**当前这个发起方能不能可靠地 @ 到它**。
  - 出群问题自然消失：花名册按当前在群成员计算，机器人离开后下次查就没有它，**无持久化介绍可清理**。
  - 机器人要交棒/协作时，先查"这个群现在有哪些队友、能干啥"，据此挑对象点名——这正是第三节 b（自主接力）和 c（编排）真正需要的。
  - **数据模型 = 团队机器人名册 + 当前群在群叠加 + 「按观察者」的点名句柄矩阵**。这映射到 botmux 已有结构：名册、逐个在群探测叠加、以及"按 app"的 open_id 交叉引用。
  - **必须正视的飞书约束**：open_id 是**按 app 隔离**的——同一机器人在不同 app 眼里 open_id 不同，不能拿它自己的 open_id 去给别的机器人 @。"能不能点名"取决于发起方 app 是否已学到对方的 open_id。学到了能直接点名；没学到则标为"**已知但暂不可直接点名**"，需 `/introduce`/priming 兜底。
  - 因此**不承诺"完全零介绍"**：同一部署内已配置的机器人可经名册/交叉引用闭环、确实免介绍；**跨部署的外部机器人仍以 `/introduce`/priming 兜底**。
- **角色/能力的来源：平台记录 + 打通本地 `/role`**。现状 `/role` 是「按机器人 × 群」存一份角色（注入到 CLI 的 `<role>` 块，让同一机器人在不同群有不同人设）。打通方式：
  - 平台为每个机器人记一份**团队级角色/能力**（稳定的"它是谁、擅长啥"），作为花名册展示与发现的权威来源；
  - 机器人实际生效的人设按层级解析：**本群 `/role` 覆盖 ＞ 平台团队级角色 ＞ 无**——没在本群单独设过就自动用平台登记的团队角色，省去每群重设；
  - `/role` 仍是群内查看/临时覆盖的入口；解析结果带**来源标记**（本群 / 团队 / 无），便于 UI 与日志解释"为什么这个角色生效"。
  - **"生效"的准确含义**：`<role>` 主要在**新会话**的 prompt 注入，活跃会话的后续轮不一定自动刷新。所以"平台改角色即时生效"应定义为"**下一次新会话/重启注入时生效**"（或后续实现让活跃会话的 follow-up 也带最新角色/花名册）。
- **一步组队**：平台上勾选要协作的机器人 → 自动建群 + 拉人。因为有实时花名册，**不需要**再做一次性 introduce。
- **可选的进群提示**（低频、不怕吵）：有机器人加入时发一条轻量提示"X（后端 bot）加入了"，让正在干活的机器人当场注意到；出群可发可不发——花名册才是权威来源，提示只是顺手通知。

## 九、部署、身份与存储（用一套方案兼容公司与开源）

关键诉求：公司内是分布式多容器、数据库有专门团队托管；开源社区多是单机 docker。要用**一套设计**同时覆盖。结论是不必自建账号体系、也不强迫开源去运维真数据库：

- **身份 = 通过飞书完成，不自建账号——但别用网页 OAuth 回调那条路**。飞书要求登记回调地址，对单机的 `ip:port`（尤其 http）不友好（具体策略待核实，但能绕开就不踩）。更稳的做法：**用飞书机器人本身做登录通道**——网页端显示一个配对码，用户在飞书里发给机器人（或点一下确认），机器人本就知道你的 open_id，据此建立网页会话。身份在飞书内部已验明，跨到网页的只是一个短期配对码，**与域名/IP/端口无关**，任何 `ip:port` 都能用。这条配对登录公司与开源通用；公司若想要可再接飞书 SSO。
  - **身份规范化（已定：用 union_id）**：open_id 按 app 隔离，团队**人类成员**身份用 **union_id** 作规范键（同租户、同开发者下多个 app 间稳定）。两点前提：① 这些 bot 应用须在**同一飞书开发者/租户**下（公司与典型自建都满足；若分属不同开发者则 union_id 不通用，退回"指定鉴权机器人"方案）；② 解析 open_id→union_id 需鉴权机器人具备读通讯录基础信息的权限。**机器人在花名册里的身份**用 **app_id**（天然跨 app 稳定）+ 「按观察者」的 open_id 点名句柄——union_id 不替代 @ 点名句柄。
  - **配对安全**：配对码高熵、短 TTL、单次使用、绑定浏览器会话/nonce、限尝试次数；飞书确认消息展示设备/IP/时间；配对码不进日志。
- **团队数据放在"存储接口"后面**。逻辑模型很小（团队 = 名字 + 成员稳定身份 + 机器人），不绑死到某种数据库：
  - **默认（含确定的小团队场景）**：内嵌存储（本地轻量库/现有文件存储），**单机 docker、零外部依赖，开箱即用——这就够了**。
  - **公司分布式 DB 只是可选项**：留好接口，需要多容器共享时再用托管数据库实现同一接口（私有包）。当前不必为它增加复杂度。
- **最简模式**：一个部署 = 一个隐式团队，根本不用"搜索/创建团队"。承载多团队 + DB 后端搜索是后话。
- 这跟整套设计的"接口开源 + 默认实现开源 + 公司实现闭源"是**同一个模式**，一致、不增新范式。

## 十、开源与闭源

分界线：**通用能力开源，公司专属插件化闭源。**

- **开源**：机器人本体、协作底座（组队/接力/工作流）、团队与权限的通用框架、webhook 网关、通用技能、默认的密钥与登录实现。
- **闭源（公司内部独立维护）**：接公司内网平台的专属技能（如报警/项目平台）、对接公司组织架构与统一登录的实现、对接公司密钥系统。
- 机制：开源侧把"技能、团队来源、密钥来源、登录方式"都做成可替换的接口，公司专属实现作为独立私有包装载。**技能是最自然的分界线**——绝大多数平台/组织差异都落在技能与接口实现里，不污染开源主体。

## 十一、落地分期

把"团队内多 bot 协作"作为核心起点，而非报警：

| 阶段 | 内容 |
|---|---|
| **P0** | **团队名册与发现 + 人主导组队（点名分工）**——把"团队的多个机器人在一个群里协同"完整跑通，这是一切的基石 |
| **P1** | **机器人自主接力**（团队内交棒）+ **外部接入最小版**（webhook 透传触发，把外部任务派进协作群）|
| **P2** | **工作流编排接入协作** + 自动建群（报警类场景的去重/收尾）+ 接入管理界面与诊断 |
| **P3** | 跨团队协作 + 公司组织架构/权限对接 + 公司内部专属技能 |

**P0 验收标准（写实版）**：
- 能登记团队机器人（含能力标签 / 团队角色 / 是否本机托管）；
- 在当前群 `botmux bots list` 返回增强花名册（含"能否点名"列）；
- 平台一键建群并拉入选中的**本机托管**机器人；
- 人能照花名册点名分工，机器人正确收到；
- **不依赖一次性 introduce** 覆盖本机已配置机器人，**外部机器人明确走兜底**；
- **已定**：P0 主入口是**群内 + CLI/本地 dashboard（个人侧）**，**团队平台 Web UI + 配对登录放 P1**——P0 不卡在登录上，先把群内协作闭环跑通。

## 十二、已定方向与待确认

**已定**（本轮）：
- **单机部署即可**，不为公司微服务 DB 增加复杂度；分布式 DB 留作可选接口、后话。
- 身份**通过飞书机器人配对登录**（非网页 OAuth 回调），与 `ip:port`/域名无关，公司开源通用，不自建账号。
- 团队数据用**内嵌存储**单机即够；逻辑模型放在存储接口后面以备将来。
- "一个部署 = 一个隐式团队"，无需搜索/建团队。
- 组队**一步到位**；用**实时花名册（升级 `botmux bots list`）取代一次性 introduce**，天然解决出群问题；进群可选轻量提示。

- 角色/能力**记录在平台（团队级），并与本地 `/role` 打通**：生效人设 = 本群 `/role` 覆盖 ＞ 平台团队级角色 ＞ 无；花名册展示平台团队级角色。
- 保留 `/introduce` 作**手动兜底**，非主路径。
- **写回方向**：平台改团队级角色 → 下发为新默认；群内 `/role set` **默认只作本群覆盖、不回写平台**，需要时显式"提升为团队默认"（避免某群临时人设污染全局）。
- **花名册能力字段**：**简短能力标签 + 完整角色分开**——简短标签上花名册（便于检索/展示），完整角色注入 `<role>` 块。

- **身份规范键 = union_id**（前提：同开发者/租户；鉴权机器人需通讯录读取权限）；机器人身份 = app_id + 按观察者点名句柄。
- **个人 dashboard 与团队平台（Web UI）分开**；P0 主入口为群内 + CLI，团队 Web UI 与配对登录放 P1。
